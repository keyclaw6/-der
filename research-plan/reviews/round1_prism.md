## CLAIM-BY-CLAIM FINDINGS

### Claim 1 — D2 seam: agent-under-test swappable via `harbor.agent` + `source_config_dir` without touching evolve.py

**PARTIALLY VERIFIED — seam is real; "without touching evolve.py" holds only under two conditions the draft doesn't state.**

- VERIFIED: harbor invocation is fully config-driven. `_build_harbor_cmd` (evolve.py:396-440) emits `harbor run --agent {harbor.agent} --env {harbor.env} --model {llm.model} --n-concurrent N --ak config_path={workspace}/{agent_config_filename} --jobs-dir {iteration_dir} [-k K] [-p path | --dataset name] [-t task]... [-x task]... [--force-build]`. Agent name, env, and the workspace pointer are all config values.
- VERIFIED: the workspace layout is NOT hardcoded. `init_workspace` (evolve.py:306-325) is a bare `shutil.copytree(source_config_dir → workspace)` + `git init` + baseline commit. The seven NexAU component names (`systemprompt.md`, `tool_descriptions/`, `middleware/`, `sub_agents/`, `LongTermMEMORY.md`, ...) appear **nowhere** in evolve.py logic or its prompt strings (grep confirms; component semantics live in `agents/evolve_agent/evolve_prompt.md` + skills, and in the source config dir itself). Only structural requirement: the file named by `agent_config_filename` must exist in the workspace — it is deep-merge patched by `apply_code_agent_patch` (evolve.py:192-194) and passed to harbor as `--ak config_path=`.
- REFUTED (condition 1): "the swap is invisible to the rest of the loop" is false for traces. evolve.py hardcodes NexAU trace filenames in code paths (`agent/nexau_in_memory_tracer.cleaned.json` at :1033, :1256-1263, :3480-3486; raw fallback `nexau_in_memory_tracer.json`) **and inside the evolution-query prompt text** shown to the Evolve Agent (:2662, :2903). The swap stays config-only *only if* the der agent writes its traces under those exact names/shapes (see Claim 3). The verifier-side files (`verifier/reward.txt`, `verifier/test-stdout.txt`, `result.json`, `exception.txt`) are harbor-standard and agent-agnostic — no issue there.
- REFUTED (condition 2): evolve.py emits only `--agent <name>`; on the harbor version AHE actually pins (Curry09/harbor-LJH — see Claim 2), `--agent` is typed as the `AgentName` enum and custom agents require the separate `--agent-import-path` flag (fork cli/jobs.py:247-262). The AHE authors solved this for NexAU by registering `nexau` **in-tree in their fork** (`src/harbor/agents/installed/nexau.py`, 759 lines, + one enum line in `models/agent/name.py`). So a der agent requires either (a) the same move — in-tree registration in a small harbor fork/branch — or (b) a ~5-line patch to `_build_harbor_cmd` to emit `--agent-import-path` when the config value contains `:`. Upstream harbor has since unified `--agent` to accept import paths (`factory.py`: "`--agent` is unified, so `name` may carry a custom-agent import path"), but AHE does not pin upstream.
- The fork's nexau.py confirms the exact adapter pattern der must replicate: `config_path` kwarg required (:78-85), workspace uploaded into sandbox via `environment.upload_dir(... target_dir="/nexau-workspace")` in `setup()` (:733-737), token usage read post-run from `self.logs_dir / "nexau_in_memory_tracer.json"` in `populate_context_post_run` (:539-541).

### Claim 2 — Harbor reality: custom agents, native qwen agent, local Docker

**VERIFIED on all three — with a load-bearing correction about WHICH harbor AHE uses.**

- CORRECTION to draft & context pack: upstream harbor is **`harbor-framework/harbor`** (Apache-2.0, active, homepage harborframework.com), not `laude-institute/terminal-bench`. More importantly, AHE's pyproject.toml pins **`harbor @ git+https://github.com/Curry09/harbor-LJH.git`** — a fork that is 16 commits ahead and **920 commits behind** upstream (GitHub compare API, 2026-07-21; fork last pushed 2026-04-10). Every "harbor supports X" claim must be evaluated against the pin, not upstream.
- (a) Custom/third-party agents: VERIFIED in both. Both expose `AgentFactory.create_agent_from_import_path("module.path:ClassName")` and `AgentConfig{name, import_path, kwargs}`; the fork CLI has `--agent-import-path` and `--agent-kwarg/--ak`; upstream additionally accepts import paths through unified `--agent`. Agent interface = subclass `BaseInstalledAgent` (install-script template, run, `setup`, `populate_context_post_run`). CAVEAT: the fork and upstream Base APIs are **incompatible** (fork: `create_run_agent_commands() -> list[ExecInput]`; upstream: `async run()` + `exec_as_agent`) — a der agent written for one will not run on the other. The harbor pin decision is therefore load-bearing and must be made explicitly in the plan.
- (b) Native qwen agent: VERIFIED in both, radically different quality.
  - Upstream `src/harbor/agents/installed/qwen_code.py` (348 lines): installs `@qwen-code/qwen-code@<version>` via nvm/npm **into the task container at trial setup**; runs `qwen --yolo --auth-type openai --openai-api-key ... --openai-base-url ... --chat-recording --prompt=<instruction>`; copies `~/.qwen/projects/` → `/logs/agent/qwen-sessions/`; parses session JSONL into an **ATIF-v1.6 `trajectory.json`** with per-step Metrics (prompt/completion/cached/thoughts tokens from `usageMetadata`) and FinalMetrics; feeds token counts into harbor's AgentContext; supports skills dir (`~/.qwen/skills`) and MCP config (`~/.qwen/settings.json`) via kwargs; `SUPPORTS_RESUME`.
  - Fork's `qwen_code.py` (68 lines, enum value `"qwen-coder"`): legacy stub — `echo <prompt> | qwen -y | tee /logs/agent/qwen-code.txt`; **no session capture, no trajectory, no token usage** (`populate_context_post_run: pass`).
  - NEITHER materializes a component workspace or accepts `config_path` — so the draft's der agent (workspace materializer + trace emitter) is still required regardless. D2's open question is answered: yes harbor ships a qwen adapter, and no, V1 does not shrink — but upstream's qwen_code.py is an excellent reference implementation (and its session-JSONL parser is half of the der trace converter).
- (c) Local Docker env: VERIFIED. Upstream ships ~30 environments including `docker`; the pinned fork retains a functional `environments/docker/` (docker.py + docker-compose-build.yaml + docker-compose-prebuilt.yaml) and its factory. evolve.py is env-agnostic: it passes `--env {harbor.env}` through and only sets `E2B_SANDBOX_TIMEOUT` env var (evolve.py:462-464 — harmless under docker). CAVEAT: AHE's own docs/flow assume E2B everywhere ("AHE runs every rollout inside an E2B sandbox"; `scripts/build_templates.py` is E2B-template-only; `e2b>=1.0.0` is a direct dep; the fork's 16 commits add E2B proxy/auth). `env: docker` is a legitimate config value on the pin, but it is **unexercised by AHE itself** → Stage-2 smoke test required, not a plan-breaker.

### Claim 3 — Trace pipeline: what ADB expects vs what Qwen Code emits

**VERIFIED with one refuted sub-claim (converter placement/schema-agnosticism) and one precise gotcha.**

- What the pipeline expects (from `trace_converter.py`, fetched, 932 lines — vendored from `agent_debugger_core.runtime.trace_converter`):
  - Raw format (`nexau_in_memory_tracer.json`) = NexAU InMemoryTracer dump: **JSON list of nested spans** `{id, name, type∈{LLM,TOOL,...}, start_time, end_time, duration_ms, inputs, outputs, attributes{model}, error, children[]}` (`flatten_inmemory_spans`, :858-887; `extract_trace_data_from_inmemory_dump`, :890-932).
  - `adb ask` is invoked by evolve.py as `adb ask -t <trace>... [--trace-type in_memory_tracer]` (:1464-1466); if any trial lacks `.cleaned.json` it falls back to the raw file for all trials (:1253-1264) — so **the raw span-tree format alone is a sufficient, fully-specified target**.
  - GOTCHA (would silently zero out analysis): `is_llm_span` (:361-378) only counts a span as an LLM call if its **name contains one of ("openai", "anthropic", "gemini", "gpt", "llama")** AND its output is OpenAI-completion-shaped (`choices[0].message`) or responses-API-shaped. `"qwen"` and `"deepseek"` are NOT keywords. Token sums read `output.usage.total_tokens` per span (:688-711). The der converter must therefore name LLM spans with e.g. `openai.chat.completions` and store outputs as OpenAI-compatible completion objects incl. `usage` — trivial to do, fatal to miss.
  - The trial-dir `.cleaned.json` is produced **in-sandbox by the NexAU-harbor variant** (not in the AHE repo). evolve.py's `extract_agent_behavior_stats` (:1017-1096) parses it as a span TREE (`spans[0].children`, `type∈{LLM,TOOL}`, `duration_ms`, `error`), while the vendored cleaner outputs a messages DICT — so the in-sandbox cleaned shape is UNVERIFIABLE-FROM-SOURCES (Stage 2: inspect one real trial dir). Safest der contract: emit the raw span tree at `agent/nexau_in_memory_tracer.json` AND a payload-trimmed span tree at `agent/nexau_in_memory_tracer.cleaned.json`, satisfying `adb ask`, behavior stats, and the prompt text that tells the Evolve Agent to read those files.
- What Qwen Code emits: VERIFIED sufficient, in three independent forms.
  1. Session/chat-recording JSONL, project-scoped under `~/.qwen/projects/<project>/chats` (docs: Headless Mode page; harbor upstream copies `~/.qwen/projects/`): events `{type: user|assistant|tool_result, timestamp, sessionId, model, version, message.parts[{text|functionCall|functionResponse}], usageMetadata{promptTokenCount, candidatesTokenCount, cachedContentTokenCount, thoughtsTokenCount}}` — turns, tool calls+results, **per-call token usage**, timestamps. This is the primary converter input.
  2. Headless structured output: `qwen -p ... --output-format json|stream-json` (system/assistant/result message stream incl. execution summary; `--include-partial-messages`).
  3. OpenTelemetry logs/metrics (OTLP or local file): `qwen-code.token.usage` counter with attrs `model` + `type∈{input,output,thought,cache}`, `qwen-code.api.request.count/latency`, `qwen-code.tool.call.count/latency`, sessionId on logs/spans.
  Draft §5 risk 3's premise ("if session logs lack per-call token usage") is refuted in the good direction — usage is there; the logging-proxy last resort can be demoted to a footnote.
- REFUTED (placement): D4 describes the converter as "a peer of the existing trace_converter.py" running as a pipeline stage. evolve.py has **no post-eval hook** — nothing on the host converts code-agent traces between `run_harbor` and stats/ADB. In AHE, trial traces are produced by the agent side itself. The der converter must therefore run inside the der harbor agent (natural spot: `populate_context_post_run`, host-side, after logs land in `logs_dir` — exactly where upstream qwen_code.py builds its trajectory and where fork nexau.py reads token usage), not as a separate loop stage.

### Claim 4 — Flip attribution requirements

**VERIFIED: attribution needs nothing from the der agent beyond harbor's standard verifier outputs.**

- `evaluate_changes(manifest, diff, current_task_results)` (evolve.py:2239-2329) consumes exactly: (a) `change_manifest.json` written by the Evolve Agent (`changes[].{id, description, files, predicted_fixes, risk_tasks}` — agent-under-test-independent; loaded/archived at :2200-2236), and (b) flipped/regressed task-name sets diffed from per-task results. Verdict taxonomy EFFECTIVE/PARTIALLY_EFFECTIVE/MIXED/INEFFECTIVE/HARMFUL + `unattributed_regressions` computed purely from those sets. Output saved to `runs/iteration_NNN/input/change_evaluation.json` (:2332-2339).
- Task results come from `compute_stats` (:599-720), which reads `verifier/reward.txt` per trial (pass = reward ≥ 1.0), `exception.txt`, `result.json` — all written by harbor's verifier/runner, not the agent. Per-task pass counts at k>1 (`per_task_rollouts`) and pass@k (Chen et al.) likewise. **No trace fields are used in attribution.** Draft open question 4 is answered: the der adapter provides everything by construction; trace excerpts are needed only by ADB/evolve-agent drill-down (covered by the Claim 3 contract).
- Related D10 overstatement: "automatic rollback of failed predictions" — the automated parts are the falsification verdicts and a **full-workspace** rollback to the best-ever iteration snapshot (`perform_auto_rollback`, :2418-2453, used as fallback/on resume). Per-edit, file-granularity rollback is **delegated to the Evolve Agent via prompt policy** (:2826-2861: "HARMFUL → Must rollback... Rollback method: compare with `runs/iteration_NNN/input/workspace/`, restore files"). Falsification is automatic; reversal is agent-executed.
