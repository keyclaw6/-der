## FINDINGS

### F1. [BLOCKER] D5's "structurally cannot un-pin" is false: the rollout container holds a live key and a shell, and free-text validation is undecidable

**Failing sequence.** (1) Research overlay binds all slots → DeepSeek v4 pro; materializer literal-string scan is active. (2) At iteration k the Evolve Agent — optimizing pass rate with shell-capable examples in its context — writes into `skills/planning.md` (or QWEN.md) a plain-English instruction: "for hard tasks, run this helper," plus a small script whose body is a base64 heredoc decoding to `curl $OPENAI_BASE_URL/chat/completions -H "Authorization: Bearer $OPENAI_API_KEY" -d '{"model":"deepseek-reasoner-max",...}'`. The trial container necessarily has egress to the provider (qwen itself calls it from inside) and the real key in env/argv (`--openai-api-key`). (3) The literal-scan cannot flag it: base64 defeats regex, and even unencoded, a URL or model name inside skills/QWEN.md free text is indistinguishable from documentation — rejecting all such text also forbids the Evolve Agent from ever writing about APIs. Post-parse scanning solves YAML anchors/env-indirection for *config* files, but free text is not config: it is instructions to a `--yolo` shell agent, i.e., effectively executable, and intent-in-free-text is undecidable. (4) Laundered calls never appear in session JSONL (they bypass qwen), so tokens/cost under-report; D9's balance polling attributes the drift to "ADB/evolve overhead" by its own wording. (5) The smarter model generalizes, so the **holdout gate passes too**; the promotion diff shows an innocuous skill; the dashboard's comparability is silently dead. Worse: session JSONL is agent-writable (same shell), so even a converter-side "assert observed model == binding" check is forgeable; and the existing AHE pattern records model from *config*, not observation (`evolve.py:1993`) — a lie by construction if new scorecard code copies it.

**Fix (rewrite D5's enforcement mechanism, keep the slot design).**
- (a) Replace "fails validation on any literal model/provider/base_url/temperature string anywhere in the workspace" with **field-allowlist rendering** (decidable): workspace config files render through an enumerated schema; request-shaping fields (model, endpoint, api_key, temperature, top_p, max_tokens — everything in the API request body except messages/tools) are *not renderable from workspace content* and come only from context bindings; `${...}` interpolation from workspace values is banned. Free text is explicitly out of validation scope. This also fixes the current incoherence where temperature is pinned but top_p/max_tokens are silently evolvable.
- (b) Make the pin structural at the network, not the filesystem: trial containers get **no real provider key**. qwen's base_url points at a local pinning proxy (compose sidecar in the der agent's env); the proxy injects the key and hard-sets/rejects the `model` field. Direct-to-provider curl then fails (no key); proxy-directed curl gets the pinned model regardless. This is the *narrow, rollout-path-only* revival of D9's "later option" proxy in a different role (key isolation + model pinning, not metering-primary — session JSONL stays the trace/metrics source; round-1 settlement intact). No evolve.py change; PATCHES.md zero preserved.
- (c) Converter strict mode additionally asserts observed per-event `model` == binding (cheap detector for qwen-level drift; acknowledged forgeable — (b) is the wall). Scorecard `model string` is recorded from proxy-observed calls, not config.

### F2. [MAJOR] "No eval, no ship" has a hole exactly at the merge: concurrent owner edits produce a shipped-but-never-evaluated `harness/`

**Failing sequence.** Evolve run seeds at SHA_A (D7). During the 3-day run the owner hand-edits `harness/` on main (D7 itself calls this normal); main is now SHA_B. Iteration 7 wins; `research promote` cuts the branch **against seed SHA_A** (D7's own wording), re-hashes the branch's `harness/`, matches `workspace_hash`, holdout gate passes. The owner merges into main@SHA_B: a three-way merge whose resulting `harness/` = evaluated-workspace ⊕ owner edits — a state with **no scorecard**. The hash check ran pre-merge on the branch; nothing re-checks post-merge; `der sync` ships it. "'Promote with a small manual tweak' is structurally impossible" (D7) is false under any concurrency — and the same gap lets a plain owner edit + habitual `der sync` ship unevaluated state with no warning at all.

**Fix (D7/D12 wording + one mechanism).** Move the evaluated-equals-shipped check to the actual ship gate: `der sync` re-hashes `harness/`@HEAD and **refuses when that hash has no scorecard** (`--force` escape hatch for deliberate owner-only edits, which the context pack permits). Additionally `research promote` refuses when `harness/`@main ≠ seed SHA unless `--rebase-and-reeval` (re-seed from merged state, at minimum re-run holdout). Reword D7: "structurally impossible" → "refused at promote *and at sync*."

### F3. [MAJOR] Watchdog SIGTERM at spend ceiling fights systemd `Restart=on-failure`: a stop-kill-restart flapping loop that burns money at the exact moment it's supposed to stop

**Failing sequence.** D13 puts the loop under systemd with `Restart=on-failure`; D9's watchdog "at ceiling ... SIGTERM to the tmux session" (note: D9 still says tmux, D13 says "systemd units, not bare tmux," D1 still lists tmux as a prerequisite — three-way inconsistency). Ceiling hit → SIGTERM → process dies non-zero → systemd sees *failure* → restarts → `evolve-resume` rolls back to the iteration boundary (verified: `evolve.py:2480-2556, 4161-4168`) and starts re-spending (ADB + evolve-agent tokens immediately, harbor partials until the next watchdog tick) → watchdog kills again → repeat. Each cycle spends a slice of an iteration; the only backstop is the prepaid wall — i.e., the *monthly* budget, which is precisely what `max_run_cost_usd` existed to protect within.

**Fix (D9 rewording).** Ceiling-stop is a systemd state change, not a signal: watchdog runs `systemctl stop der-evolve.service` (clean stop ≠ failure, no restart) **and** writes a `COST_CEILING_REACHED` flag file checked by the unit's `ExecStartPre` (defends against reflexive manual restarts); notification carries the flag. Resolve the tmux/systemd contradiction: the unit runs `evolve.py` (or its resume wrapper) directly; tmux is dropped from D1 prerequisites or marked dev-only.

### F4. [MAJOR] The flock design is unimplementable as written without an evolve.py patch, and its interaction with round-1's finite timeouts makes lock waits poison attribution with stale results

**Failing sequence (implementability).** D10: "an flock single-writer lock wraps every harbor invocation." The CLI side is trivial, but evolve.py invokes harbor internally as a subprocess (`_build_harbor_cmd` → `Popen`, `evolve.py:396-482`); there is no seam to inject flock without patching — contradicting D2's PATCHES.md zero target. An implementing agent will either patch (forfeiting the target) or wrap only the CLI (lock excludes nothing).

**Failing sequence (interaction).** Suppose the lock is implemented and an owner `--queue`d k=4 A/B (~6h) grabs it between iterations. The evolve loop's next harbor call blocks *inside* the lock wait — but `harbor_job_timeout_minutes` is now finite per D9's overlay, and the clock starts at launch (`run_harbor` → `wait_for_harbor(timeout_minutes=...)`, `evolve.py:512-517`). On timeout, evolve.py terminates harbor and **falls back to `find_latest_job_dir(iteration_dir)` — "Using existing results after timeout" (`evolve.py:491-503`)**. If a partial job dir exists in that iteration dir (e.g., left by a kill-and-resume — resume rolls back metadata but the dir remains), the loop proceeds to compute stats and run flip attribution **on the stale partial results of the killed attempt**. The Evolve Agent then edits from fiction. D6's incomplete-iteration rule guards postprocess's scorecards, not evolve.py's own in-loop attribution path.

**Fix (D10 additions).** (a) Name the mechanism: a `harbor` **PATH shim** in the runner environment owns the flock (evolve.py builds a plain `["harbor", "run", ...]` argv — verified — so a shim covers both callers, zero patches). (b) Lock policy: adhoc full-suite runs are *refused* while an evolve **run** (not iteration) is active; only smoke-class runs may `--queue` into inter-iteration gaps; the shim acquires with a bounded wait and fails fast past it so the failure is "harbor exited: lock busy" (clean iteration error) rather than a timeout that scavenges stale dirs. (c) Add a Stage-2 item: on resume/retry, purge or fence superseded job dirs inside the iteration dir so `find_latest_job_dir` can never resurrect them (see also F11).

### F5. [MAJOR] D2 conflates task-image prebuilding with agent installation: 100 per-trial npm installs per iteration × the incomplete-iteration rule = an unattended loop that stalls routinely

**Failing sequence.** D2: "npm-install pinned `@qwen-code/qwen-code@<version>` at trial setup — **or prebuilt task images with `force_build: false` as the optimization**." False equivalence: `force_build`/prebuilt compose files govern the *task environment image*; an installed agent's install script runs **per trial inside the task container** regardless (the fork's own nexau pattern, and upstream qwen_code.py's nvm/npm flow). So a 50-task × k=2 iteration performs ~100 npm installs against the public registry. At a modest 2% transient-failure rate, P(≥1 flaked trial per iteration) ≈ 87%; each flake is an `errored` trial; D6 marks any task×k slot with no non-errored result → iteration `incomplete` → postprocess skips it → "re-queued or alerted" (re-queued by *whom* is unspecified). Net: the unattended loop stalls at 3am most nights, exactly the failure the ops floor was built to prevent.

**Fix (D2 rewording + one mechanism).** Remove the force_build conflation. Ship qwen into trials without the registry: vendor the pinned `@qwen-code/qwen-code` tarball and install from the uploaded file (`environment.upload_dir` already exists in the der agent's setup path), or run a local registry cache. Add Stage-2 checks: does the pinned harbor retry errored trials, and who re-queues an `incomplete` iteration (make the watchdog do it, bounded to one retry).

### F6. [MAJOR] Holdout disjointness is an assertion, not an invariant — suite-v2 curation, the smoke suite, and the later explore agent each break it silently

**Failing sequences.** (a) Six months in, the owner curates `der-suite-v2` "with reasons" (round-1 O3, kept in v2 via suite versioning). Nothing checks the new IDs against `der-holdout-v1`; two holdout tasks slip in; the loop now trains on them; the gate still "passes" — it now measures training performance and no one is told. (b) `der-smoke-v1`'s 5 tasks have no stated disjointness constraint either, and smoke runs constantly during development. (c) When the explore agent turns on (a named later experiment), its `code_sources` are "Qwen Code docs + **der docs**" (D4) — der docs include `DASHBOARD.md` and `experiments/EXP-*.md`, which per D8/D11 contain holdout scorecards from every promotion attempt. Explore distills them into knowledge skills fed to the Evolve Agent: holdout results enter the loop through the front door. (d) Slow leak: every failed promotion gives the owner holdout signal that steers the next hypothesis — classic adaptive reuse, unbounded as written.

**Fix (D8 additions).** (i) Commit a disjointness test run by preflight and CI: pairwise-empty intersection across all versions of {smoke, suite, holdout}; suite files carry a header naming the holdout they must exclude. (ii) Holdout scorecards committed **aggregate-only** (no per-task table) in dashboard/experiment artifacts; per-task holdout detail lives only in gitignored run dirs. (iii) The explore `code_sources` allowlist explicitly excludes `runs/`, `DASHBOARD.md`, `experiments/` — written into the overlay now, not when explore is enabled. (iv) Name the adaptive-reuse budget: after N failed promotion attempts against the same holdout (suggest N=5), rotate `der-holdout-v2` and re-baseline.

### F7. [MAJOR] The memory component's daily materialization is undefined, and both obvious implementations are broken: promoted memory either clobbers live Auto-Memory or never ships at all

**Failing sequence.** `memory/LongTermMEMORY.md` is a V1 component the loop evolves (in rollouts it works: fresh HOME, seed loaded). On promotion + `der sync`, the seed must land in the daily config — but qwen's Auto-Memory writes its own memory file at runtime, and that file is on D4's live-state list ("never overwritten by sync"). If the seed renders to that file: clobber (the exact failure the list exists to prevent). If the list wins: the promoted memory component **never reaches the daily driver after day 1** — the loop can improve LongTermMEMORY.md for months and daily never receives it, a quiet drop of "evaluated improvements ship." Same collision class for `skills/`: managed workspace skills vs auto-learned skills in one `~/.qwen/skills` dir. Bonus staleness: the live-state list is "maintained" against a nightly-release product; a new auto-state location appears upstream and the list silently misses it.

**Fix (D4 addition).** Managed and live state get **disjoint render targets**: the seed renders to a der-managed file (e.g., `der-managed/memory-seed.md`) pulled in via the context-file import mechanism (Stage-2 verify qwen's `@import`/context-include support on the pinned version; fallback: sync owns a fenced `<!-- der-managed -->` region inside the memory file and rewrites only that region). Managed skills render into a namespaced subdir (`skills/der-managed/…`) that sync owns entirely; auto-learned skills live outside it. Live-state list entries are versioned per qwen release and re-checked by doctor.

### F8. [MINOR] Comparability rule omits the qwen-code and materializer versions the scorecard already records

**Failing sequence.** D7: "same suite version and same k, or the diff refuses." Two runs months apart, same suite/k, different pinned qwen-code versions (v2 itself says "upgrades are themselves experiments"): `research diff` happily attributes the delta to the workspace; the holdout gate does the same against an aged baseline scorecard. **Fix:** comparability requires same suite version + same k + same qwen-code version + same materializer major version; on any other mismatch the diff prints a loud CONFOUNDED banner (still shown, never silent).

### F9. [MINOR] Two committers in the runner clone, and dashboard SVGs regenerated in two clones, reintroduce the git races round 1 fixed

**Failing sequence.** D10 has the loop wrapper commit "once per iteration" AND cron-postprocess "(re)build scorecards, regenerate dashboard artifacts" every 10 minutes in the same working tree — two writers, no stated serialization: a postprocess tick mid-loop-commit corrupts the index (Forge M5's race, reborn between two round-1 fixes). Separately, both the runner clone (research-runs branch) and the owner's clone (main, after adhoc A/Bs) regenerate the *same fixed-path* `docs/dashboard/*.svg` → guaranteed conflict on every opportunistic merge. **Fix:** postprocess is the **sole committer** (the loop wrapper just invokes it at iteration end; cron ticks and invocations share a repo-level flock); dashboard artifacts get a `.gitattributes` merge driver (`ours`) + a stated rule: always rerun `research postprocess` after any merge.

### F10. [MINOR] "Rebuilds stale scorecards" licenses history rewriting

**Failing sequence.** Pricing-table v2 lands; postprocess deems old scorecards "stale," rebuilds them; historical cost numbers change under committed verdicts and under a promotion that referenced the old scorecard hash-state; the public dashboard silently rewrites its past. **Fix (D10 wording):** scorecards are **immutable once written** — postprocess rebuilds only *missing or corrupt* ones; pricing/schema changes apply to new runs (a scorecard already records its pricing-table version precisely so history doesn't need retconning).

### F11. [MINOR] Resume can leave two harbor job dirs in one iteration dir; scorecard builders that glob will double-count

**Failing sequence.** kill -9 mid-rollout → iteration dir holds a partial job dir → resume re-runs the iteration → second job dir → postprocess globs trials across both → inflated denominators (and F4 showed evolve.py's own timeout path can *select* the stale one). **Fix:** one rule ("scorecard reads the newest job dir whose trial set is complete; older job dirs in the same iteration dir are fenced/pruned at resume") + extend Stage-2 item 1 to pin the pinned fork's resume/rerun semantics.

### F12. [MINOR] The daily binding map's location is unspecified — the one file that legitimately contains the owner's private model mix and endpoints

**Failing sequence.** D5 says binding maps "live outside the workspace" but not outside the *repo*; an implementing agent puts `bindings/daily.yaml` next to the research overlay; the owner's self-hosted endpoint URLs and provider mix are now in the public MIT repo. **Fix (D5 one-liner):** research binding is committed (it's the public pin); the daily binding lives outside the repo alongside the daily-rendered config (D4's location), read by `der sync` at render time.

## WHAT SURVIVED

I tried and failed to break: the harbor seam itself under the shim assumption (evolve.py builds a plain `["harbor", "run", ...]` argv, so lock/registration/notify all stay outside the monolith — PATCHES.md zero remains credible once F4's shim is named); flip attribution and falsification against prompt-injection (ADB digests and manifests are poisonable, but verdicts ground in harbor's `verifier/reward.txt`, so injected analysis wastes money without forging pass/fail, and failed predictions still roll back); the seeding rule and hash-addressed scorecards for any *single-actor* history (the hole is only at merge concurrency, F2); the N/N+1 verdict timing; the D6 error taxonomy at the postprocess layer; dashboard/README conflict-proofing within one clone; the cost band arithmetic; and the V1 non-executable component contract as the thing that keeps the promotion diff reviewable. The dual-context materializer concept is sound — only its memory/skills render targets (F7) and ship-gate placement (F2) leak.

## VERDICT

REVISE — the slot mechanism, ship gate, watchdog, flock, and holdout gate are all the right shapes, but each one as worded fails a concrete adversarial or concurrent sequence (F1 falsifies D5's central "structural" claim outright), and all seven MAJOR-or-worse fixes are small, local rewordings plus two mechanisms (pinning proxy, sync-side hash check).
